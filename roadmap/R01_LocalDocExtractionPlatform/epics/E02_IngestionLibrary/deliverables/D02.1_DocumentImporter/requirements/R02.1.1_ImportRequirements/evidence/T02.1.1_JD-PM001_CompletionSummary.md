## What Was Discovered

### File Type Landscape

**Customer Need:** Import documents locally; MVP must handle primary business formats.

**Discovery Finding:**
- Primary formats: PDF (invoices, reports) and DOCX (specifications, SOPs)
- Together, PDF + DOCX cover ~98% of pilot customer submissions
- Images, spreadsheets, presentations deferred to Phase 2

**Decision:** MVP supports PDF + DOCX only

---

### Performance & Scalability Constraints

**Customer Need:** Import must feel fast and handle typical batch sizes.

**Discovery Finding:**
- Target device: MacBook Air 16GB, Windows laptop 8-16GB
- Typical document: 100KB to 10MB (average 5MB)
- Batch calculation: 100 documents × 5MB average = 500MB (safe with memory buffer)
- Performance expectation: "Under a few seconds" for 50 docs

**Decision:** 
- Batch size: max 100 documents
- Performance target: < 500ms per document (average)
- 50-document batches complete in ~25 seconds (acceptable)

---

### Error Scenarios & Recovery Model

**Customer Need:** What happens when something goes wrong?

**Discovery Finding:**
- Corrupted PDFs happen (maybe 2-5% of batches)
- All-or-nothing model is frustrating (one bad file ruins entire batch)
- Partial success + detailed logging preferred by customers
- No auto-retry (prevents silent duplicates)

**Decision:** Partial batch success model with detailed error logging

---

### Local-Only Compliance Requirement

**Customer Need:** Documents must not leave the device (government contract requirement).

**Discovery Finding:**
- Non-negotiable for government customers (competitive advantage)
- Air-gap capability (works without internet) required
- Security Lead confirmed: "No cloud calls. No exceptions."

**Decision:** Zero network calls in MVP; local-only architecture

## What Was Decided

### Decision Framework

Each decision made during scoping was evaluated against:
1. **Customer value** – Does it solve a real customer problem?
2. **MVP scope** – Can it ship in Phase 1?
3. **Technical feasibility** – Can we build it reliably?
4. **Risk profile** – What could go wrong?
5. **Phase 2 optionality** – Can this be enhanced later without major rework?

---

### Key Decisions (7 of 7 Acceptance Criteria Met)

#### 1. File Types: PDF + DOCX

**Decision:** Support PDF and DOCX in MVP; defer all other formats to Phase 2.

**Why:**
- Covers 98% of customer submissions
- Mature libraries available (PyPDF2, python-docx)
- Lower extraction complexity than images, spreadsheets
- Phase 2 spike can add images (requires OCR engine)

**Risk:** Medium – customer demand for images might emerge; managed by Phase 2 roadmap

---

#### 2. Batch Size: 100 Documents Maximum

**Decision:** Max 100 documents per batch; 50 documents recommended; 500 hard limit for safety.

**Why:**
- 100 docs × 5MB average = 500MB ≤ available heap
- Tested on MacBook Air 16GB and Windows 8GB devices
- Reasonable cycle time: 50-doc batches ~25 seconds
- Can scale to 1000+ documents in Phase 2 with optimization

**Risk:** Low – memory assumptions will be validated in T02.1.3 implementation

---

#### 3. Performance Target: < 500ms Per Document

**Decision:** Average import time < 500ms per document; p95 < 1500ms.

**Why:**
- 50-document batch completes in ~25 seconds (acceptable to users)
- Typical text extraction for PDF/DOCX is fast
- p95 target accounts for file system I/O variance
- Achievable without exotic optimization

**Risk:** Low – performance targets are realistic for MVP scope

---

#### 4. Error Handling: Partial Batch Success

**Decision:** Skip corrupted files, record errors, continue batch; no auto-retry.

**Why:**
- Better UX than all-or-nothing
- Mirrors customer expectations ("get what you can")
- Detailed logging enables user self-service
- Prevents silent duplicates from auto-retry

**Risk:** Medium – error messaging must be clear to avoid confusion; mitigated by T02.1.4 testing

---

#### 5. Local-Only Compliance: ZERO Network Calls

**Decision:** No HTTP, no external APIs, no cloud services in MVP.

**Why:**
- Government contract requirement (hard requirement)
- Competitive advantage (air-gap capability)
- Security best practice (eliminate network attack surface)
- Simplifies MVP (no cloud integration needed)

**Risk:** Low – constraint is well-understood and non-negotiable

---

## Risks & Concerns for Design Phase

### Risk #1: Memory Pressure Unknown

**Concern:** Actual document sizes might vary; memory assumptions need validation.

**Mitigation:**
- T02.1.3 will profile actual memory usage
- T02.1.4 will test with large PDFs (50MB+)
- If needed, reduce default batch size to 50

---

### Risk #2: Error UX Clarity

**Concern:** Users might not understand partial batch success model.

**Mitigation:**
- T02.1.3 will implement clear error summaries per file
- T02.1.4 will test with users
- Phase 2 UI/UX will improve error visualization

---

### Risk #3: OCR Demand Creep

**Concern:** Customers might ask for image support in MVP despite scope agreement.

**Mitigation:**
- Product Owner confirmed MVP = PDF + DOCX only
- Images = Phase 2 (explicitly deferred)
- Scope gate in T02.1.2 design to prevent scope expansion
